# 人脈手札 — 持續優化指南

這份是您之後反覆參考的工作手冊。每次想優化前，回來看一次。

---

## 核心：四步優化迴圈

```
①觀察 → ②決策 → ③實作 → ④發佈
  ↑                              ↓
  └──────────  使用者回饋  ←──────┘
```

- **觀察**：誰在用？卡在哪？哪些功能根本沒人碰？
- **決策**：這週只做 1–3 件事，挑「最多人抱怨」或「自己用最痛」的
- **實作**：開 Cowork 找 Claude，描述問題，讓我改檔案
- **發佈**：測試 → 上傳到 Cloudflare → 通知使用者

---

## 必備基礎設施（先架好這四個再開始迭代）

### 1. 版本備份

#### 工作資料夾結構

```
人脈資料庫/
├── deploy/
│   └── index.html                 ← 目前線上的版本（最新）
├── 版本備份/
│   ├── index-2026-05-13-原始.html
│   ├── index-2026-05-20-加了PWA.html
│   └── ...
└── 持續優化指南.md（本檔案）
```

#### 規則

- **每次重大修改前**，先把 `deploy/index.html` 另存一份到 `版本備份/`，檔名加日期和重點：
  ```
  index-2026-05-20-加了PWA.html
  index-2026-05-25-修正OCR bug.html
  ```
- 改壞了，從備份複製回 `deploy/index.html` 重新上傳即可。

#### Cloudflare 自帶的版本歷史（雙重保險）

每次您上傳到 Cloudflare，它都會保留一份。要回復：

1. Cloudflare → Workers & Pages → 您的專案
2. 點 **Deployments** 分頁
3. 找到想回復的那一版 → 點右邊 **⋯** → **Rollback to this deployment**

### 2. 使用者回饋管道

光靠口耳相傳會嚴重低估問題。建議至少做一個：

#### 選項 A：Line @ 官方帳號（在地最方便）
- 申請 Line 官方帳號（免費）
- 把 Line ID 放在 App 的「設定」頁
- 朋友直接 Line 您

#### 選項 B：Google Forms（最省力）
- forms.google.com 建一個 5 題回饋表
- 把連結放在 App 裡（例如設定頁加一行「回報問題」）
- 答覆會自動進 Google 試算表

#### 選項 C：App 內回報按鈕
- 直接在 App 加一個「📝 回報問題」按鈕
- 點下去打開 mailto: 或 Line URL
- 跟 Claude 說「幫我加回報按鈕，回報內容寄到 xxx@gmail.com」即可

---

### 3. 用量數據 — Cloudflare Web Analytics

**免費、5 分鐘設定、不拖慢網站、符合 GDPR。**

#### 設定步驟

1. Cloudflare → Workers & Pages → 您的專案
2. 上方分頁找 **「Web Analytics」**（或 Settings → Analytics）
3. 點 **「Enable」**
4. 完成

之後就可以看到：

| 指標 | 用途 |
|------|------|
| Page views | 每天多少人打開 |
| Unique visitors | 多少不同人 |
| Top countries | 主要從哪個國家 |
| Top referrers | 從哪個連結來的（Line/Facebook） |
| Average session | 平均停留多久 |

> ⚠️ Web Analytics **不會看到**個別使用者的資料、不會追蹤行為（這是它的優點），所以不會違反隱私。

### 4. 想法清單

開一個 `待辦清單.md` 在資料夾根目錄。建議格式：

```markdown
# 待辦清單

## 🔥 馬上做（這週）
- [ ] Bug：切到西方後手機版字太小
- [ ] 加上回報問題按鈕

## 💡 想做（下個月）
- [ ] 加 PWA，可以加到主畫面
- [ ] 名片 OCR 改用後端代發 Key

## ❄️ 也許（之後再說）
- [ ] 雲端同步
- [ ] 多語言版

## ✅ 已完成
- [x] 2026-05-13 修正東西切換不會自動轉換（佐證 v3）
```

**用法：**
- 自己用時想到什麼就丟到「馬上做」或「想做」
- 朋友回饋時也記到對應欄位
- 每週看一次「馬上做」，挑 1–3 件跟 Claude 一起做
- 做完打勾搬到「已完成」（這也是您自己的成就感來源）

## 工作節奏

| 頻率 | 做什麼 |
|------|--------|
| **每天** | 自己用一下，發現不順手就記到清單 |
| **每週**（推薦週日 30 分鐘） | 看 Web Analytics + 回饋訊息，更新清單，挑 1–3 件 |
| **每 1–2 週** | 開 Cowork 實作 → 測試 → 上傳 → 通知朋友 |
| **每月** | 回看數據：使用者真的需要這些功能嗎？該不該砍掉沒人用的？ |

## 兩個容易踩到的雷

### 雷 1：IndexedDB 結構不能亂改

您的人脈資料存在每個使用者的瀏覽器（不在您伺服器上）。如果某次更新改動資料結構（例如改了欄位名稱、拿掉了某個欄位），使用者已經存的資料會讀不出來。

**保護方法：**
- 每次跟 Claude 提需求時，如果想改資料儲存方式，**明確說出來**：「這次需要動到資料儲存結構」
- Claude 會幫您寫遷移邏輯（讓舊資料自動轉換成新格式）
- 也會在發佈前提醒使用者「請先匯出備份」
- 一定要保持 App 裡的「匯出 JSON」按鈕可用，那是使用者的逃生門

### 雷 2：不要追求一次完美

您最大的優勢是「可以每週更新」。
- 一次更新一點，使用者跟得上，您也不會壓力大
- 看到別人的產品很豐富，那是好幾年累積，不是一次做完
- 80/20：花 20% 時間改的小細節，可能解決 80% 使用者的抱怨

## 何時該升級到「進階版」工作流？

目前流程（拖檔上傳）適合：使用者 < 500 人、更新頻率 < 一週一次。

如果有一天到達這些信號，再考慮進階：

| 信號 | 該升級什麼 |
|------|-----------|
| 每週要更新 3 次以上 | 學 GitHub Desktop + Cloudflare 連 GitHub 自動部署 |
| 使用者超過 500 人，開始有人說「換手機資料就沒了」 | 考慮加後端做雲端同步 |
| 有人說「我覺得這值得付費」 | 規劃免費版+付費版 |
| 一週收到 10+ 個回饋訊息 | 換成 Notion / Linear 管理待辦 |

不到信號就不要硬升級，免得自找麻煩。

## 給未來自己的提醒

- 使用者**不會主動讚美**，但**會主動抱怨**。沒被抱怨的功能不代表好用，可能只是沒人用。
- 「想做」清單會越來越長，這正常。重點不是清空它，是每週挑出最重要的。
- 自己用得不爽的功能，**99% 朋友也會不爽**。優先修自己的痛點。
- 介面的「禮貌」很重要：載入提示、錯誤訊息、操作成功的回饋，這些細節決定觀感。
- 每一次發佈都是學習機會。發出去之後觀察、收回饋、調整、再發。這就是做產品的核心。
