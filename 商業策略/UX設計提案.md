---

## 1. 稽查方法 + 涵蓋範圍

**方法**：閱讀 `index.html` markup（行 1169–1365 為 UI 結構）+ JS handler（行 2032–3045）+ 對照 `產品路線圖.md` 的 Phase 1 目標，逐項走完使用者四階段旅程。沒有跑真機點擊，但對單檔 HTML 而言，code 即 UI。

**涵蓋**：
- ① 診斷階段：onboarding 選東/西 → 自我命理 sheet → self-welcome 個人畫像 → 引導加第一筆
- ② 識人階段：FAB ＋ quick-add bar / 完整 add sheet / 名片 OCR / 編輯
- ③ 記錄階段：detail 頁的命理畫像、合作分析、觀察、筆記、深入識人 AI
- ④ 服務階段：人脈列表、搜尋、五行/元素 chip 篩選、設定頁、回饋/匯出

**不在範圍**：CSS 細節（交給 ui-designer）、命理算法正確性（交給 fortune-engine）、Worker 後端（交給 deployer）。

## 2. 整體觀察（5 句話）

1. **核心引擎已扎實，但「使用者第一次到第三次打開」之間的成長曲線斷裂**：第一次有引導，但回訪沒有 progress feedback、沒有「下一步推薦做什麼」的勾子。
2. **資訊架構偏「列出來」而非「指出來」**：列表照新增順序排，沒有「最近互動」、「需要 follow-up」、「久沒見」這類使用者實際決策時想要的視角。
3. **詳情頁資訊密度極高，但缺少「scan-to-action」**：命理畫像、合作分析、深入識人三大塊堆疊 1000+ 字，使用者在見面前 30 秒打開難以「掃一眼就能用」。
4. **設定頁是個雜物抽屜**：命理/API/資料/回饋四個性質完全不同的事擠在同一頁，新使用者找「我的命理」要捲過 API Key 區塊。
5. **「服務階段」幾乎缺席**：第 ④ 階段（找出該聯絡誰、跨人脈比較）目前只能靠搜尋框 + 五行 chip 達成 20% 的需求，這是後續最大的價值缺口。

## 3. 四階段旅程稽查

### 3.1 ① 診斷階段（onboarding + 自我設定）

#### 流程現況（line 1169-1198, 2032-2135, 2684-2752）
1. 首次開啟 → onboarding 全螢幕：東/西雙卡 → 點一張 → 淡出 0.6s
2. 自動彈出 `editSelf(true)` self sheet（line 2042），標題「歡迎！先設定您自己」
3. 填年月日（東方）或月日（西方）→ 保存
4. `showSelfWelcome()` 顯示「這就是您 — PERSONALITY PROFILE」+「建立第一個人脈 →」CTA

#### 觀察 + 痛點

| # | 觀察 | 為什麼是痛點 | 優先 |
|---|---|---|---|
| 1.1 | onboarding 兩張卡的 tag「八字 / 數字易經 / 五行」對新使用者是專有名詞（line 1180-1182） | 不信命理的人看到這 3 個詞已經想關掉。需要「先賣價值，再講方法」 | 🔥 P0 |
| 1.2 | 點完東/西沒有任何「為什麼選這個」的解釋，直接跳 self sheet | 使用者剛承諾用一個 framework，App 沒給回饋確認「我選對了」 | 💡 P2 |
| 1.3 | self sheet 標題「歡迎！先設定您自己」（line 2687）— 但 sheet 本體只有四個 input 框，沒有解釋「為什麼要填我自己」 | 使用者填完不知道下一步看到什麼，焦慮 | ⚠️ P1 |
| 1.4 | self-welcome 頁複用 detail-section 的密集排版（line 2056-2111），對「人生第一次看自己命盤」的使用者太重 | 第一次 wow moment 應該是「一句話 + 一個視覺亮點」，現在直接給五行分佈條 | ⚠️ P1 |
| 1.5 | 「稍後再說」/「建立第一個人脈 →」（line 2122-2123）只有兩個選項 — 沒有「先逛逛 App」 | 對「不想立刻填別人資料」的使用者只能走「稍後」，但其實 App 主畫面是空的 | 💡 P2 |
| 1.6 | onboarding 副標題 `Choose Your Way of Seeing People`（line 1173）是英文 | 對台灣使用者第一眼陌生，且和產品「中文 first」原則衝突 | 💡 P2 |

#### 建議

- **1.1** 把 tag 改成「結果導向」的詞：東方卡顯示「適合：合作模式 / 性格判讀 / 衝突預警」、西方卡顯示「適合：性格速判 / 國際朋友 / 直覺解讀」。把「八字」這種詞放在點進去後才出現。**成本：低**（只改 markup）
- **1.3** self sheet 上方加 30 字解釋條：「先填您自己 → 之後加的每個人都會跟您做合作配對分析」。讓使用者懂為什麼這步是必須的。**成本：低**
- **1.4** self-welcome 改成 3 段式：① 一句話 hero（例「您是 庚金 · 身強 — 行動派、邏輯先行」）② 一張視覺卡（五行雷達或一張代表色塊）③ CTA。其餘細節摺疊在「看完整畫像」連結後面。**成本：中**（要重寫 `renderSelfFortuneAnalysisHTML` 的第一屏）

### 3.2 ② 識人階段（新增 / 匯入 / 編輯）

#### 流程現況（line 1284-1340, 2458-2615）
- **三個入口**：
  - FAB（line 1284）+ 號 → 完整 `addSheet`（七大區塊：名片 OCR、基本、命理、信任、頻率、特質、筆記）
  - 頭部 ⚡ icon（line 1215）→ `quickAddBar`（line 1286-1291，只有姓名 + 生日 2 欄）
  - 名片 OCR：addSheet 內第一個區塊（line 1302-1308），需 API Key 才能用（line 2620）

#### 觀察 + 痛點

| # | 觀察 | 為什麼是痛點 | 優先 |
|---|---|---|---|
| 2.1 | FAB（+）和頭部 ⚡（quick-add）功能重疊，使用者不知道差別 | 兩個入口都叫「新增」但行為不同。⚡ 是 productivity feature，新使用者根本不會碰 | ⚠️ P1 |
| 2.2 | addSheet 預設展開 7 個區塊（line 1302-1334），垂直長度 > 2 屏 | 手機使用者在咖啡廳填名片時要捲半天才找到「保存」 | 🔥 P0 |
| 2.3 | 名片 OCR 區擺在 sheet 最上方但需要 API Key 才能用（line 2620），沒填 Key 點下去只 toast「請先填 Key」 | 等於告訴 80% 使用者「這裡是裝飾」。應該動態隱藏 OR 改提示「免費試用 OCR 需要 Key，去設定看看」 | 🔥 P0 |
| 2.4 | 「八字 vs 數字易經」要在命理區自己切 tab（line 1327, 2196）；新使用者不知道「兩個都可以填」（提示在 line 2198 但只有 11px 灰字） | 引導沒做夠。多數人會以為要二選一 | ⚠️ P1 |
| 2.5 | quick-add bar 生日格式 `19900315`（line 1288）— 沒有自動格式化（不能填 1990/03/15、1990-03-15） | 使用者打字會想加分隔符，被擋住會挫折 | ⚠️ P1 |
| 2.6 | 沒有「批次匯入」入口在主畫面 — 匯入埋在設定頁底部（line 1264） | Phase 1 朋友圈試用，他們有現成 contacts/Notion/Google Contacts，第一次用想倒一批進來 | 💡 P2 |
| 2.7 | 編輯人脈只能走「進詳情 → 點編輯」（line 2438）。在列表上長按 / 滑動沒有 quick action | 桌機使用者習慣 right-click，手機使用者習慣 swipe | 💡 P2 |
| 2.8 | 互動頻率 `<select>`（line 1332）只有「每週/每月/每季/每年/少於每年」 5 個值，沒有「最近一次見面日期」 | 「頻率」是 framework、「最近一次」才能驅動「該聯絡誰了」的服務階段。資訊架構錯位 | 🔥 P0 |
| 2.9 | 「性格特質」要使用者用逗號分隔自己打字（line 1333） | 沒人會手打「理性, 直接, 行動派」這種東西，做成 chip 多選會用爆 | 💡 P2 |

#### 建議

- **2.1** 移除 ⚡ quick-add 或改名「快速記名」。在 FAB 上加長按手勢觸發快速模式（如果手機平台支援）。**成本：低**（隱藏一個按鈕）
- **2.2** addSheet 分階段：必填只有「姓名 + 生日」 → 點「儲存並繼續補充」展開後續區塊。或所有「主觀觀察」「筆記」做成 accordion 預設摺疊。**成本：中**
- **2.3** 名片 OCR 區塊未配置 Key 時換成「💡 上傳名片自動填欄位（需設定 API Key）」灰底 hint，直接點開設定頁。**成本：低**
- **2.4** 命理區上面加「☯ 兩種都可以同時填」的視覺更強的 hint（badge 而非 11px 文字）；或在第一次新增時引導使用者「先試試其中一個」。**成本：低**
- **2.8** **新增「上次見面」欄位**（date picker 或 quick-pick：今天/上週/上個月/3 個月前）。儲存後 detail 顯示「距上次 23 天」。這是 ④ 階段所有功能的基石。**成本：中**（資料 schema + 列表排序）

### 3.3 ③ 記錄階段（深入識人 AI、筆記、互動頻率）

#### 流程現況（line 2274-2453, 2854-3037）
- detail 頁固定順序：hero → 聯絡方式 → 命理畫像 → 合作分析 → 觀察 → memo → 深入識人 → action bar
- 深入識人按鈕（line 2438）位於最底部 action bar，3 agent 接力（命/合/話）
- 沒填 Key 走 proxy / 填 Key 用 Sonnet

#### 觀察 + 痛點

| # | 觀察 | 為什麼是痛點 | 優先 |
|---|---|---|---|
| 3.1 | detail 頁總長度極高（命理畫像 + 合作分析 + 深入識人滿開展開 ≥ 2000 字） | 「見面前 30 秒打開查」這個 use case 完全沒被服務。應該分 tab 或預設摺疊 | 🔥 P0 |
| 3.2 | 「深入識人」CTA 埋在最底部 action bar（line 2438） | 這是產品最大亮點 + 唯一 AI 功能，應該放在 detail 上半段 | 🔥 P0 |
| 3.3 | 深入識人完成後顯示 3 個 agent 卡（命/合/話 line 2860-2893），但所有 ul 是平鋪 | 「下次見面」「可問問題」這種 actionable 內容應該獨立浮出，不是長文 ul | ⚠️ P1 |
| 3.4 | 沒有「複製深度分析摘要」按鈕（待辦清單已列為💡） | 使用者想分享或存到別處只能截圖。Phase 2 P0 | ⚠️ P1 |
| 3.5 | 筆記欄是 textarea（line 1334）— 一次見面寫一段，沒辦法做「2026-03-12 跟他吃飯討論了 X」的時間軸 | 重度使用者抱怨：寫多了找不到。對「服務階段」也是大缺口 | 💡 P2 |
| 3.6 | 語音輸入結果直接 append 到 textarea（line 2658），沒有區分段落或時間戳 | 同 3.5 | 💡 P2 |
| 3.7 | 沒填命理資料的人脈，detail 頁直接缺一大塊（沒有 placeholder 提示「沒填生日 → 補上就能跑命理」） | 使用者忘了自己沒填，以為 App 壞了 | ⚠️ P1 |
| 3.8 | 深入識人 cache 失效時顯示「資料已變更 · 重新分析」（line 2858）超低調的 underline 文字 | 應該更明顯 — 但「重新分析」會吃 API 配額（免費版每日 3 次），這個 UI 也要傳達成本 | 💡 P2 |
| 3.9 | 深入識人 error 顯示 message 但沒區分 rate-limit / 網路錯 / parse 錯 | 使用者撞到 rate-limit 不知道是「明天再試」還是「我做錯什麼」 | ⚠️ P1 |

#### 建議

- **3.1 + 3.2** detail 頁重組：第一屏只放 `[hero + 一句深入識人摘要 + 大 CTA: 深入識人 / 重新分析]`，下面 3 個摺疊卡：「命理畫像」「合作分析」「我的筆記」。預設都收起。**成本：中**（改 `showDetail` 的 HTML 組合邏輯）
- **3.3** 深入識人結果顯示時，「對話顧問 — 可問 / 避開 / 下次見面」這塊獨立到頁面頂部當「下次見面 cheatsheet」。**成本：低**
- **3.5** 筆記改 timeline 形式：每次新增是一個 entry（日期 + 文字）。**成本：中-高**（schema + UI 都要動，留 Phase 2 做）
- **3.7** 沒命理資料的 detail 頁顯示「💡 補上生日就能跑命理分析 [編輯]」inline 卡片。**成本：低**
- **3.9** error toast 對 429/limit 顯示專門訊息（「今天免費額度用完了，明天再試 / 設定 API Key 解除上限」）。**成本：低**

### 3.4 ④ 服務階段（找出人脈、合作分析、跟誰互動）

#### 流程現況
- 搜尋框（line 1222-1226）匹配姓名/別名/領域/角色/公司/特質/筆記
- 五行/元素 chip 篩選（line 2168-2188）+ 相生/相剋
- 列表預設按 contacts 陣列順序（= 新增順序，line 2253-2265），沒有 sort UI
- 沒有跨人脈比較 / matrix view / 月報

#### 觀察 + 痛點

| # | 觀察 | 為什麼是痛點 | 優先 |
|---|---|---|---|
| 4.1 | 列表沒有「排序」選項（按互動頻率、最近見面、信任度、新增日期） | 服務階段的所有問題都從這個能力長出來。**沒排序 → 沒洞察** | 🔥 P0 |
| 4.2 | 沒有「我這個月該聯絡誰」的智慧推薦面板 | Phase 1 朋友試用時，使用者打開 App 沒事可做，會流失 | 🔥 P0 |
| 4.3 | 列表上每筆 ci-right 顯示 coop mode 或 trust dots（line 2264），但不顯示「上次互動」、「身份」 | 列表掃一眼資訊密度不足，使用者要點進去才看得到 role | ⚠️ P1 |
| 4.4 | filter chip「相生 / 相剋 / 挑戰」對新使用者太抽象 | 應該改「合得來 / 互補 / 注意點」這種白話 | ⚠️ P1 |
| 4.5 | 沒有「合作 matrix」（路線圖 Phase 2 也許做）— 但即使只是「跟某個人脈比較另一個」也沒入口 | 比較是 CRM 核心動作，目前完全缺席 | 💡 P2 |
| 4.6 | 沒有「生日提醒」/「久未聯絡提醒」— 路線圖 Phase 1 也許做 | 把使用者拉回 App 的最強鉤子 | ⚠️ P1 |
| 4.7 | 沒有 export 單一人脈的「合作備忘卡」（PNG / Markdown）給朋友看 | 跟 Phase 2 的「分享深度分析」是同一件事，但個人卡更輕量 | 💡 P2 |

#### 建議

- **4.1 + 4.3** 列表上方加「排序」下拉（最近見面 / 信任度 / 新增時間 / 名字），同步把 ci-right 改成「最近見面：X 天前」當主資訊。**成本：中**（需要 2.8 的「上次見面」欄位先做）
- **4.2** **首頁加「今日聚焦」卡片**（top of list）：① 久未聯絡 Top 3（>頻率 × 1.5 倍時間沒見） ② 本月生日。**成本：中**，但這是讓 App 從「填資料工具」變「日常打開工具」的關鍵跳板
- **4.4** chip 文案改白話：「全部 / 木 火 土 金 水 / 合得來 / 要小心」**成本：低**
- **4.6** PWA notification API 暫不做（複雜）；先在 App 內當天打開時顯示 banner。**成本：低**

## 7. 不該做的事（避免功能蔓延）

對齊 `產品路線圖.md` § 不會做的事 + 我自己的判斷：

| ❌ 不要做 | 為什麼 |
|---|---|
| **群組分析 matrix（>3 個人交叉比較）** | 服務階段使用者真正想要的是「該找誰」，不是「3 個人誰跟誰合」。後者是好玩但少用。Phase 3 之後再評估 |
| **生日 / follow-up 推播（push notification）** | PWA push 在 iOS 17+ 才支援，受眾小、實作成本高、容易煩人。先用 App 內 banner 就夠 |
| **筆記富文本編輯（粗體/列表/圖片）** | textarea 已經夠用。富文本會把單檔 HTML 維護成本拉高 |
| **多語系（i18n）** | 目前單一台灣使用者，西方主題只是命理切換不是語言切換。Phase 3 才考慮 |
| **AI 自動分類人脈 / 自動寫筆記** | 把「使用者主動判斷」的核心動作交給 AI 會稀釋產品定位（命理當 framework，不是 AI 當算命師） |
| **匯入 Google Contacts / LinkedIn / 名片夾** | 違反「資料留本機」原則 + 引入 OAuth 維護成本。JSON 匯入已經夠用 |
| **行事曆同步** | 路線圖明確列為「不做」 |
| **使用者帳號 / 雲端同步**（Phase 3 前） | 路線圖明確 — 隱私第一，先用 JSON 匯出 + 之後 E2EE |
| **付費牆 UI**（在 Phase 2 退出條件達成前） | 商業評估明確 — 100 使用者前所有模型都虧錢，先用「贊助意願」按鈕測試 |
