> 撰寫者：product-manager subagent

## TL;DR（3 句話）

**現在不要上邀請碼**。Phase 1（active users < 30、目標朋友圈試用）上邀請碼只會把唯一的留存燃料（口碑 + 低摩擦試用）燒掉，且當前 rate limit + KV 已經把成本風險擋住了。**建議的觸發時機是 M4–M5「增長啟動」階段（同步上 Lifetime 限量買斷時）**：採用**用途 E（A+D 組合）「Lifetime 早鳥碼」+「推薦人 referral 碼」雙軌**，技術走 **Worker + KV 存碼狀態 + 過期/限額**，形式採 `RENMAI-XXXX` 4 碼自訂尾碼（贊助者 / KOL 可分辨來源）。在那之前先做兩件「邀請碼前置」零成本準備：① URL `?ref=xxx` 純記錄 referrer 寫進 KV（不影響使用流程）② Lifetime 預購名單 email 收集。

## 1. 為什麼需要邀請碼（先界定問題）

在動手設計形式之前，先問「不做會死嗎」。把可能要解的問題列出來：

| # | 想解的問題 | 現況怎麼處理 | 沒邀請碼會死嗎 |
|---|---|---|---|
| Q1 | 防止公開網址被陌生人灌爆，把 Anthropic 帳單燒爛 | Worker 已做 3 次/IP/日 + 200 次/全站/日 + 1500 token 上限 + origin 檢查 | 不會。現有 rate limit 已是 hard cap，最壞案例月成本 NT$80/IP，全站日上限 NT$60。**這個風險已被擋住** |
| Q2 | 保持封閉式 beta、品質可控 | 朋友圈邀請靠口頭傳 URL | 不會死，但會慢。Phase 1 北極星只要 30 註冊，手動發 URL 完全夠 |
| Q3 | 追蹤誰邀請了誰（病毒擴散驗證、未來推薦獎勵發放） | 完全沒做。Web Analytics 只能看流量、看不到「A 介紹 B」 | Phase 2-3 會死。`產品路線圖.md` Phase 2 驗證假設明確列「病毒擴散會發生 → 看 referrer 流量比例」。沒有 ref 機制就驗證不了 |
| Q4 | 區隔「免費 free rider」vs「Lifetime 付費者 / KOL 合作」的進階功能 | 還沒有付費功能上線 | M4 起會死。Lifetime 賣出後要怎麼讓買家「永久解鎖無限深入識人」？要嘛靠 email 對清單（很爛），要嘛靠兌換碼（合理） |
| Q5 | 朋友抱怨「我都用完 3 次了想多看」但又不想填 API Key | 目前的解法是「填你自己的 Key」 | 不致命。但這是 Phase 1 → 2 之間最常見的卡點。1-2 個 trusted 早期使用者給「20 次/日」鉤子很便宜 |
| Q6 | KOL / Lifetime 換內容時要給對方「比一般人多的權益」 | 沒處理過 | M3 起會死。`變現執行藍圖_12個月.md` 階段 2 M2 W3 明確要「給 KOL Lifetime 換內容」，那 Lifetime 怎麼發？就是邀請碼 |

**結論**：
- Q1（防爆）**不需要**邀請碼。
- Q2（封閉 beta）**不需要**邀請碼（規模太小）。
- Q3、Q4、Q6 **需要**，但只在 M3+ 才開始需要。
- Q5 是 **bonus**，可以順便做。

**問題的真實本體**：「不是『要不要邀請碼』，而是『M4 上 Lifetime + 推薦獎勵時，要用什麼機制兌換 / 追蹤』」。把邀請碼當成 Phase 2-3 的**基礎建設**規劃比較對。

## 2. 五種可能用途逐項評估

### A. Gate 全 App（沒碼進不來，封閉式 beta）

| 維度 | 評估 |
|---|---|
| 對齊產品路線圖 | ❌ 違反 Phase 2「公開軟啟動」目標（要 100-200 註冊 + SEO 流量）。封閉 beta 等於宣告「SEO 流量來了會撞牆」 |
| UX 影響 | 嚴重負面。SEO 訪客點進來看到「請輸入邀請碼」轉身就走。沒 onboarding 機會、沒口碑 |
| 技術成本 | 中等。要在 Worker 加 middleware 擋所有 `/`、要記住「已驗證」狀態（cookie / localStorage）、要處理 PWA 重開驗證 |
| 風險 | 一旦解禁就回不去（拆掉門檻會被酸「之前是裝高級嗎」）。SEO 計分被毀（爬蟲看到 gate） |
| **推薦？何時？** | **❌ 不推薦，任何時候都不要**。產品定位是「個人 CRM 個人使用」，封閉 beta 不適合 |

---

### B. Gate 進階功能（深入識人無上限 / Sonnet 升級 / 雲端備份）

| 維度 | 評估 |
|---|---|
| 對齊產品路線圖 | ✅ 完全對齊。Phase 3「Cloud Service NT$59/月」、競品調查修正版「Cloud Sync NT$99/月」都需要「誰可以用」的識別機制 |
| UX 影響 | 正面。原本要綁帳號才能買 Pro，現在「一張碼兌換 = 解鎖」沒帳號摩擦（符合「資料留本機」的隱私訴求） |
| 技術成本 | 中等。Worker 多一個 `/api/redeem` endpoint + KV 存碼狀態 + 前端記住「我是付費者」（IndexedDB `_local:premiumCode`） |
| 風險 | 碼洩漏 = 該名額永久損失。要做「single-use」+「限定 device 第一次兌換才生效」雙保險 |
| **推薦？何時？** | **✅ 強烈推薦，M4-M5 起**。這是 Lifetime / Cloud Sync 的兌換通道，無此不行 |

---

### C. Bypass rate limit（有碼免費版額度從 3/日 → N/日）

| 維度 | 評估 |
|---|---|
| 對齊產品路線圖 | 半對齊。Phase 1「朋友圈試用」階段，朋友抱怨「3 次不夠」是常見痛點 — 這時候 trusted 早期使用者一張 20 次/日 鉤子很便宜（一個朋友 200 NT$/月成本天花板）。但不對齊「先驗證留存」原則：把鉤子發太早會分不清「他是因為 quota 才留下、還是真的喜歡」 |
| UX 影響 | 正面。但 trade-off：發碼變成「我自己當客服在收申訴」，無法 scale |
| 技術成本 | 低。已有 KV `rl:date:ip` 機制，多加「`il:carrier`」白名單 + IP 多檢一個碼欄位即可 |
| 風險 | 一張碼若被多人傳閱會被一群人共用。**必須綁 device 或時間** |
| **推薦？何時？** | **△ 條件性推薦，M1-M3 之間做為「朋友圈精準鬆綁」的小工具**。發 5-10 張，私訊朋友「這碼一個月內 20 次/日」。M4 後因為有 Lifetime 取代，這個用途自動退場 |

---

### D. Referral 追蹤（誰邀請了誰，未來給原邀請者獎勵）

| 維度 | 評估 |
|---|---|
| 對齊產品路線圖 | ✅ 完全對齊。`產品路線圖.md` Phase 2 明確要驗證「病毒擴散會發生」；`競品調查與商業路線修正` M4-M5 計劃明確要「推薦一人購買 Lifetime → 推薦人得 NT$200 信用點」 |
| UX 影響 | 中性偏正。被推薦的訪客自然帶 `?ref=xxx` 進來，沒額外摩擦；推薦人在設定頁看「我推薦了 N 人、得 X 點」很有成就感 |
| 技術成本 | 低（純記錄）到中（要計算獎勵）。M3 前可以只記錄（KV 寫一筆 `ref:date:ip = code`），M5 起再做積分結算 |
| 風險 | 推薦人自己用兩台手機 fake referral — 但金額不大（NT$200/筆），自我裂變即使被作弊也沒到要設防的程度。**真正風險是隱私**：要追到「誰邀請誰」就要存一些 PII（至少 email），這跟「資料留本機」原則有張力 |
| **推薦？何時？** | **✅ 推薦，分兩階段做**。M3：先做 `?ref=xxx` 純流量歸因記錄（KV 計數即可、不存 PII）。M4-M5 Lifetime 上線時加「兌換時填推薦碼 → 推薦人 email 通知獎勵」 |

---

### E. 組合方案（最有可能採用的）

把 B + D 串起來，C 當小工具備用：

**E.1**：**Lifetime 兌換碼** + **referral 連結追蹤** + **少量 trusted 朋友 rate-limit 白名單碼**。一套 KV 系統 + 一套發碼工具同時服務三個用途。

| 維度 | 評估 |
|---|---|
| 對齊產品路線圖 | ✅ 完整覆蓋 M3-M12 所有商業節點 |
| UX 影響 | 正面，因為對使用者來說「有碼貼進去就解鎖」是最低認知負荷的兌換動作 |
| 技術成本 | 中。Worker + KV + 前端兌換 UI 一次做完，三個用途共用 |
| 風險 | 一旦 KV bind 出包（被刪、被改 region）所有付費者瞬間斷線。要有匯出 backup |
| **推薦？何時？** | **✅ 最終推薦方案。M3 末做 referral，M4-M5 加 Lifetime 兌換 + trusted 白名單** |

## 3. 技術方案對比

| 方案 | 安全性 | 一次性成本 | 維護成本 | 適用階段 | 結論 |
|---|---|---|---|---|---|
| **純前端驗證**（碼寫死在 index.html、JS 比對） | 極低，view-source 1 分鐘破解 | 0.5 hr | 0 | 任何階段都不行 | ❌ 不要 |
| **純前端 + obfuscation**（碼經 hash 對照表） | 低，被 reverse engineering 兩小時破解 | 1 hr | 0 | M0-M1 朋友圈絕對信任時可暫用 | △ 不建議 |
| **Worker + 環境變數白名單**（碼存在 Worker secret 裡） | 中。碼不可見、但每加碼要 redeploy | 2-3 hr | 加碼/撤碼都要改 secret + redeploy | M2 試水溫期，碼數量 < 20 | △ 過渡可用 |
| **Worker + KV 存碼狀態** | 高。碼隨時新增、狀態可變 | 6-8 hr（含 admin endpoint） | 加/撤碼用 wrangler CLI 或 web 介面 | M3+ 常駐方案 | ⭐ 推薦 |
| **Worker + KV + 過期/限額/single-use 完整生命週期** | 高+ | 10-14 hr | 同上 | M4+ Lifetime 上線時 | ⭐⭐ 最終目標 |
| 帳號系統（D1 / Supabase + email 註冊登入） | 最高 | 30+ hr | 持續客服 | M9+ 才考慮 | ❌ 違反「本機儲存」原則，Phase 3 才評估 |

**選用邏輯**：第一版（M3 末）走「Worker + KV 存碼狀態」(6-8 hr)，第二版（M4-M5）增量加上過期/限額/single-use（再 4-6 hr）。**完全跳過純前端方案**，因為 view-source 破解的 PR 災難不值得省那 6 小時。

## 4. 邀請碼的「形式」設計

### 4.1 字元 / 長度

| 候選 | 範例 | 評估 |
|---|---|---|
| 隨機 6 碼 alphanumeric | `K7M2P9` | 短、好念、好打。但無法分辨來源 |
| 隨機 8 碼 alphanumeric | `K7M2P9XQ` | 衝突空間夠大（36^8 ≈ 2.8 兆），無法暴力枚舉 |
| 前綴 + 4 隨機 | `RENMAI-K7M2`、`KOL-JANE`、`LIFE-A1B2` | **可分辨來源**、長一點但有品牌感。**推薦** |
| UUID | `f47ac10b-58cc-...` | 太長，使用者打不下去（雖然可以複製貼上） |
| 自訂字（給 KOL） | `JANE2026`、`LU-CONSULT` | 高辨識度，但碰撞風險高、需個別協調 |

**推薦**：用前綴分用途。

- `RENMAI-XXXX` (4 碼隨機)：一般推薦 / 朋友圈試用
- `KOL-XXXX` 或 `KOL-{nickname}`：KOL 合作碼（一張限多次兌換）
- `LIFE-XXXX`：Lifetime 買斷者兌換（一張限單次兌換）
- `FRIEND-XXXX`：trusted 朋友 rate-limit 白名單（綁 device）

> 視覺一致性：全大寫 + dash 分隔，跟 `RENMAI` 品牌一致。

### 4.2 單次 vs 多次

| 用途 | 單次/多次 | 為什麼 |
|---|---|---|
| Lifetime 兌換 | 單次 | 一張碼 = 一個人付了 NT$890 |
| KOL 合作 | 多次（限額 N） | KOL 推 100 個粉，給 50 個解鎖 |
| Referral 個人碼 | 多次（無限） | 推薦多少人都行，計數累積成積分 |
| trusted 白名單 | 單次（綁 device） | 防止傳閱 |

### 4.3 過期

| 用途 | 過期策略 |
|---|---|
| Lifetime | **永不過期**（這是 Lifetime 的定義） |
| KOL | **發出後 90 天兌換有效，兌換成功後綁 device 永久** |
| Referral | 推薦人積分永不過期；被推薦人「30 天內兌換有效」（防舊連結垃圾流量） |
| trusted 白名單 | **單月有效**（從兌換日起 30 天） |

### 4.4 一個碼一個人 / 一個碼多人

照 4.2 講過了。原則：**碼的「單次/多次」與用途綁定，不要事後改規則**。

### 4.5 兌換後本機 cache vs 每次 ping

| 方案 | 優點 | 缺點 |
|---|---|---|
| 本機 cache（IndexedDB `_local:redeemedCode` + `_local:tier`） | 離線可用、UX 順 | 篡改 IndexedDB 可以 fake premium。但因為 server-side `/api/deep-read` 還是會驗，篡改只能騙 UI 不能騙 API |
| 每次 ping 驗證 | 安全 | 沒網路就退級、體驗差 |
| **混合：本機 cache + 每次 deep-read 帶碼** | UI 即時、API 保護 | **推薦** |

**運作**：兌換成功時 Worker 回 `{tier:"lifetime", quota:"unlimited"}`，前端寫 IndexedDB；之後呼叫 `/api/deep-read` 時帶 `redeemedCode` header，Worker 在 KV 查狀態決定走哪個 rate limit。離線時可以開 App、看現有資料，但 deep-read 一定要連線 — 這跟現況一致。

## 5. 推薦方案

### 5.1 用途

**E.1：B + D + C 三軌共用一套基礎建設**。

- B（Lifetime 兌換）：M4-M5 上線，主目的
- D（Referral 追蹤）：M3 末上線，純記錄；M5 起加積分
- C（trusted 白名單）：M2-M3 之間順手做，發 5-10 張給朋友

### 5.2 技術

**Worker + KV 完整生命週期**。

- KV namespace：`INVITE_CODES`（新增，與既有 `RATE_LIMIT` 分開）
- 每張碼的 KV value 結構（JSON）：

```
{
  "tier": "lifetime" | "kol" | "friend" | "referral",
  "issuedAt": "2026-06-01T00:00:00Z",
  "expiresAt": null | "2026-09-01T00:00:00Z",
  "maxRedeems": 1 | 50 | -1,
  "redeemedCount": 0,
  "redeemedDevices": ["device-xxx"],
  "issuedTo": "雷蒙 / KOL Jane / friend Wang",   // 內部備註
  "referrerCode": null | "RENMAI-AB12"          // 若是被某張 referral 帶來
}
```

- Worker 新增 endpoint：
  - `POST /api/redeem`（前端傳碼 + deviceId，Worker 驗證後寫入 redeemedDevices）
  - `GET /api/code-status?code=xxx`（前端開 App 時可選查詢）
  - **不做** admin 管理 endpoint — Phase 2 規模 < 200 張碼，用 `wrangler kv:key put` CLI 手動加減就好

### 5.3 形式

`RENMAI-XXXX` 4 碼隨機尾碼 + `KOL-{nick}` / `LIFE-XXXX` / `FRIEND-XXXX` 三個變體。

### 5.4 流程（使用者視角）

**Lifetime 買斷流程**（M4 起）：

1. 落地頁點「立即購買 Lifetime」→ 跳 Lemon Squeezy / 綠界結帳頁
2. 付款成功 webhook 觸發 → Worker 生成 `LIFE-XXXX` 寫進 KV → 寄 email 給買家「您的兌換碼是 LIFE-XXXX」
3. 買家回 App，設定頁點「兌換碼」→ 貼上 → toast「✓ 已解鎖 Lifetime」
4. 之後所有 `/api/deep-read` 帶這張碼 → Worker bypass rate limit

**Referral 流程**（M3 末起）：

1. 設定頁顯示「我的推薦碼：RENMAI-AB12 [複製連結]」（每個 device 自動生成一張）
2. 朋友點連結 `https://renmai.app/?ref=AB12` 進 App → Worker 寫 `ref-visit:date:AB12 += 1`
3. 朋友若之後買 Lifetime → 結帳頁有「推薦碼」欄位帶 AB12 → Worker 算「AB12 推薦成功 +1」
4. M5 起加積分頁，推薦人可用積分換贊助回饋（咖啡券 / Lifetime 折扣）

**Trusted 白名單流程**（M2-M3 試水溫）：

1. 您手動跑 `wrangler kv:key put` 生成 `FRIEND-XXXX`
2. Line 私訊朋友「這碼 30 天內每天 20 次深入識人，貼到設定頁兌換」
3. 朋友兌換 → KV 寫 redeemedDevices → 之後該 device 過 deep-read 走加高 quota 路徑

## 6. 上線觸發條件

不要因為「規劃寫了」就馬上實作。每個用途各有獨立觸發：

| 觸發條件 | 動作 |
|---|---|
| Phase 1 結束 + 進入 Phase 2 公開軟啟動（M2-M3 之交） | 做 D 的純記錄版（`?ref=xxx` → KV 計數），1-2 hr 工 |
| 一週內 ≥ 3 個朋友 Line 抱怨「3 次不夠」 | 做 C 的 trusted 白名單（手動發 5-10 張），3-4 hr 工 |
| Lemon Squeezy / 綠界商品頁開好、準備發 Lifetime（M4 W1） | 做 B 完整 KV 系統 + redeem UI，6-8 hr 工 |
| Lifetime 賣出 ≥ 30 名 | 開 D 的積分頁面（推薦獎勵發放），3-4 hr 工 |
| KOL 合作正式開始（M2 W3 或更晚） | 個別生成 `KOL-{nick}` 碼，無新 dev 工，KV CLI 加 1 筆 |

**反觸發**（這些情況就不上）：

| 反觸發 | 為什麼 |
|---|---|
| Phase 1 active users < 10 連續 4 週 | 連使用者都沒有，邀請碼是表演 |
| 自然 referral（透過口頭 / Line 截圖 URL）已 > 5 人/週 | 不需要科技解決方案 |
| Lifetime 還沒準備好金流（綠界 / Lemon Squeezy 沒對接） | 沒有付費入口，B 用途無意義 |

## 7. 不該做的事 / 風險

### 7.1 違反路線圖的做法（不要做）

- ❌ **用邀請碼 gate onboarding**（用途 A）→ 違反 Phase 2「公開軟啟動 + SEO 引流」
- ❌ **要求註冊 email 才能用 App**（哪怕只為了 referral）→ 違反「資料留本機」隱私訴求
- ❌ **把邀請碼當成「付費版」的代名詞，全功能上鎖**（Phase 2 退出條件達成前）→ 違反商評「100 users 前不上付費牆」
- ❌ **賣 Premium 邀請碼**（碼本身當商品賣）→ 是 Lifetime 兌換碼，碼不該獨立賣，務必綁實際商品

### 7.2 技術 / 資安雷

- ⚠️ **不要把碼比對寫在前端**。即使是「我的推薦碼是 KOL-JANE」這種看似公開的碼，比對都要在 Worker 做
- ⚠️ **不要用流水號當碼**（`RENMAI-0001`、`RENMAI-0002`）— 可被枚舉，攻擊者寫腳本掃過所有號碼
- ⚠️ **KV 寫入有 eventual consistency**（最壞 60 秒）— 兌換高峰時若雙重兌換，要靠 `KV.put({ifNotExists})` 或單調計數器處理 race
- ⚠️ **不要把 `redeemedDevices` 列表存無限長**。一張多次碼若被 10000 個 device 兌換，KV value 會超過 25MB 上限。加 `maxRedeems: 50` 強制截斷
- ⚠️ **碼洩漏 / 撤銷機制**：要有「黑名單」機制（KV `revoked:RENMAI-AB12 = true`），不是只能加碼不能砍碼

### 7.3 商業 / 心態雷

- ⚠️ **不要把「發邀請碼」變成 founder 客服日常**。一張碼 10 分鐘協調對話 × 50 個朋友 = 8 小時。一上線就要有「自助索取連結」（贊助 NT$50 自動發碼），不要全人工
- ⚠️ **不要承諾「Lifetime 包含未來所有功能」太絕對**。寫法：「包含目前所有 Pro 功能 + Cloud Sync 以外的未來功能」。Cloud Sync 必須月訂閱，否則一張 NT$890 換永久雲端對冷啟動公司是自殺
- ⚠️ **不要在朋友圈試用階段就拿邀請碼當「社交貨幣」誇耀**。Phase 1 唯一目標是 5 個朋友重複使用 + 5 條痛點回饋。任何讓朋友覺得「我在被當測試品」的儀式都會降低回饋誠實度

## 8. 給使用者的下一步

### 8.1 如果您批准「最終推薦方案」（E.1：B + D + C 共用系統）

執行順序（按 M0 是現在算）：

| 時點 | 動作 | 誰做 | 估計工時 |
|---|---|---|---|
| **現在 → M2 之前** | 不做。專心 Phase 1 P0（已有的待辦清單前 4 項） | — | 0 hr |
| **M2 W1**（公開軟啟動開始時） | 落地頁 + `?ref=xxx` 純流量歸因。KV 寫 `ref-visit:date:code += 1`，前端不顯示 | deployer + ui-designer 接力 | 2-3 hr |
| **M2-M3 之間（看朋友反應）** | 視 Q5 痛點頻率決定是否做 C trusted 白名單 | deployer | 3-4 hr |
| **M3 W4 / Lemon Squeezy 商品開好前** | Worker `/api/redeem` + KV `INVITE_CODES` namespace + 設定頁兌換 UI | deployer + ui-designer | 6-8 hr |
| **M4 W1 Lifetime 上線** | 結帳 webhook → 自動生 `LIFE-XXXX` + 寄 email。配合金流商選定 | deployer | 4-5 hr |
| **M5 起（Lifetime 賣 ≥ 30 名）** | Referral 積分頁、推薦回饋發放邏輯 | deployer + ui-designer | 4-6 hr |

**總投入估計**：跨 M2-M5 約 20-25 hr，分成 5 個 sprint，每個 sprint 之間都有「不做也活得下去」的逃生口。

### 8.2 如果您只想做最便宜的一步（最小可行）

- 現在 → 5 分鐘：在 worker.js 加一個「IP 白名單常數」，把您自己 + 3 個重度試用朋友的 IP 加進去（從 Cloudflare Analytics 抄），bypass rate limit。**這不是邀請碼，但達到 C 用途的 80%**。等 M3 再升級成正式系統。

### 8.3 如果您決定完全不做邀請碼

- 也合理。配套：M4 Lifetime 上線時改用「email 對清單」（買家信箱寫入 KV，前端登入時填 email 比對）。**缺點是侵犯隱私 + 體驗差**，但開發成本省一半。

## 9. 一句話結論

> **邀請碼不是 Phase 1 的事**。現在做 = 浪費。**M2 末順手做 `?ref=xxx` 流量歸因（2 小時零成本），M3-M4 之間 Lifetime 商品準備好時，做完整的 Worker + KV 兌換系統**。形式用 `LIFE-XXXX / KOL-XXXX / FRIEND-XXXX / RENMAI-XXXX` 四類前綴，一套後端服務 B + D + C 三個用途。在那之前每談一次「要不要上邀請碼」都先問：**「現在哪個觸發條件達成了？」**達不到就再等一週。
