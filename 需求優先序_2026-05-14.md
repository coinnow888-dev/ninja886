## TL;DR（3 句話）

1. **6 項裡只有 1 項（#5 API Key 隱碼）+ #6 的 bug 修正應該插隊到本週**；其他 4 項都該排到 Phase 2 之後或直接不做。
2. **#1 通訊錄匯入** 牴觸路線圖「資料留本機」原則；**#2 每日運勢**、**#4 屬性說明書** 是把產品從 framework 拉向算命網站，違反產品定位；**#3 改收費結構** 在 active users < 100 的現階段是過早優化。

## 逐項評估（6 個）

### 1. 導入手機通訊錄 / Gmail 通訊錄

- **對齊產品路線圖**：❌ 違反「資料留本機 / 隱私第一」、且「整合 LinkedIn / Facebook」已在不會做的事清單，邏輯類似（OAuth + 第三方資料源 + 維護成本）
- **對齊商業評估**：❌ 引入 OAuth = 後端必要 = 跟 Phase 3 才該動的雲端架構提前綁定
- **UX 影響**：高（一鍵倒入 N 筆，但後續整理成本更高 — 多數通訊錄條目沒有「合作關係 / 觀察 / 命理需要的生日」，會把列表灌滿「半成品人脈」反而稀釋核心價值）
- **實作成本**：高（Google OAuth + iOS Contacts API + 欄位 mapping + 重複偵測 + UI）
- **風險**：把產品定位從「精挑細選的人脈框架」變成「全通訊錄管理工具」，跟 Notion/Dex 等老牌競品正面對撞
- **建議優先級**：❌ 不做（至少 Phase 3 前不做）
- **理由**：使用者真正的痛是「填一個人脈要 2 分鐘」（UX設計提案 #2.2），不是「我有 500 個聯絡人沒倒進來」。先解決前者。退而求其次可考慮 P2「vCard 匯入單筆」 — 但連這個都不在 Phase 1 範圍

### 2. 每日運勢功能增加用戶黏著度

- **對齊產品路線圖**：❌ 路線圖把產品明確定位為「命理當 framework，不是預測未來的算命網站」（產品定位第 12 行）
- **對齊商業評估**：⚠️ 每日運勢可能增加 DAU 但提高的是「算命依賴度」，會把潛在使用者裡「不信命理但想要 CRM 框架」那一群嚇跑（路線圖關鍵不確定性 #1）
- **UX 影響**：中（增加打開頻率，但是錯的打開理由 — 來看運勢不會用核心功能）
- **實作成本**：中（八字日干流年計算 / 星座每日運勢資料源 / 通知）
- **風險**：把產品從「個人 CRM + 命理框架」拉向「占卜 App」，跟產品定位線徹底相反
- **建議優先級**：❌ 不做
- **理由**：UX設計提案已經給出**真正會提升黏著度的方案** — 4.2「今日聚焦」卡片（久未聯絡 Top 3 + 本月生日），這是讓使用者「有事可做」的鉤子。每日運勢是「給使用者看的東西」、今日聚焦是「給使用者做的事」，後者才會留人。如果真想做，做給「自己」一個人就好（self detail 頁上加流年小卡，零成本），不要全 App 推

### 3. 深入識人改逐次收費 + 月吃到飽

- **對齊產品路線圖**：⚠️ Phase 2 才該驗證收費假設（路線圖第 102-108 行）。現在 active users = 0
- **對齊商業評估**：⚠️ 商業評估明確推薦路徑是「Phase 1 全免費 → Phase 2 上 B 贊助 → Phase 2.5 限量買斷 → Phase 3 才上 D 混合」（商評第 240-244 行）。逐次 + 月吃到飽是模型 A/D 變體，跳過了「驗證使用者是否願意付費」這個前提
- **UX 影響**：中（多一個付費牆 UI 會嚇跑試用者）
- **實作成本**：高（金流串接 Stripe/Lemon Squeezy + 訂閱狀態 + Worker 端配額管控 + 退款流程 + 發票）
- **風險**：在沒驗證「有人願意付」之前先做收費系統，等於用 10 小時工時換 0 收入的機率高
- **建議優先級**：❄️ 之後再說（Phase 2.5+，active users ≥ 100 + 贊助收入連續 3 個月 > NT$3000 後）
- **理由**：商業評估已經給出精準的觸發點 — **連續 3 個月贊助收入 > NT$3000 才上 Lifetime，Lifetime 賣出 50 名 + 月使用者 ≥ 500 才上模型 D**（商評第 313-316 行）。現在連 Phase 1 退出條件（10 個非親友 active）都還沒達到。「告訴我想法」按鈕（#6）就是要替這個決定收資料的，先看數據再決定收費模型

### 4. 不同屬性人如何相處的功能說明書

- **對齊產品路線圖**：⚠️ 介於合理（命理 framework 的延伸）和危險（變成算命百科）之間
- **對齊商業評估**：⚠️ 跟核心收費功能「深入識人」會競食 — 使用者看了通用說明書就不需要付費跑個別分析
- **UX 影響**：低-中（多一頁可看內容，但不是「打開 App 就想去的地方」）
- **實作成本**：中（要寫 5 行 × 5 行 = 25 組對照內容，或星座 12 × 12 = 144 組；內容品質要高才有用，否則就是網路抄來的常識）
- **風險**：① 內容如果是 AI 寫的會像 ChatGPT generic content，使用者馬上識破 ② 變成靜態知識庫，跟「個人 CRM」定位漂移
- **建議優先級**：💡 想做但晚點（Phase 2 中期，且要重新設計切入點）
- **理由**：使用者真正會用到的不是「金 vs 木 通用相處說明」，而是「**我**（庚金）跟**他**（甲木）相處」— 而這就是現有「合作分析」+「深入識人」在做的事。建議的替代：在合作分析區塊下面加 1-2 句「為什麼這樣判」的微說明（例：「金剋木：您主導性強，他喜歡自由 — 給空間讓他發揮，您負責收尾」），不另開頁面。**這個微改動 = 低成本 P2**

### 5. API Key 隱碼且不能修改

- **對齊產品路線圖**：✅ Phase 1 P0 級別的信任改善
- **對齊商業評估**：✅ 跟「OCR 也走 Worker proxy」方向一致（商評推 Phase 2 P0），讓 API Key 變「進階使用者選項」而非首要設定
- **UX 影響**：中-高（解 UX設計提案 X.3 + X.5 — 新使用者看到 API Key 區塊會困惑）
- **實作成本**：低（input type=password + 顯示 `sk-***xxxx` 遮罩 + 加「清除」按鈕、不要「修改」按鈕；要改就清除後重填）
- **風險**：「不能修改只能清除重填」對使用者算多一步，但相比安全/UX 的好處划算
- **建議優先級**：🔥 這週做（插隊進 UX設計提案的 P0）
- **理由**：① 安全：避免使用者開設定頁時旁人偷瞄 ② UX：解決「填了 Key 之後設定頁永遠裸露 sk-xxx」的不安感 ③ 對齊 UX設計提案 X.5「API Key 提示太長」的精簡方向。實作極簡：`<input type="password" readonly>` + 「清除」按鈕。**強烈建議併入這週 P0**

### 6. 「告訴我想法」按鈕 bug + 文案改寫

#### BUG 診斷

**結論：不是 bug。程式邏輯正確。**

- `recordPricingInterest()` 在 index.html 行 2809-2819
- 邏輯流程：`dbGet meta/pricingInterest` → `count + 1` → `dbPut` → `toast('已記錄您的意願 · 感謝')`
- toast 函式存在於行 2027，2.5 秒淡出
- dbGet/dbPut 跟其他地方（如 `apiKey`、`theme`、`self`）寫法完全一致，schema 也對得起來（meta store keyPath: 'key'，行 1974）

**「點了沒反應」最可能的原因**：
1. **toast 沒被看到** — 2.5 秒過短、或位置在螢幕底部被使用者忽略 / 設定頁本身有 scroll，toast 可能不在當前視野
2. **IndexedDB 在 iOS Safari 斷線** — 5/14 才修的 auto-reconnect bug 邊緣 case，但這種情況 toast 還是會跑（dbPut 失敗會 throw，但這個 function 沒 try/catch）
3. **dbPut 失敗 silent** — 行 2813 沒包 try/catch，如果 dbPut reject 整個 async function 就靜默掛掉，toast 不會跑。**這是潛在 bug**

**修法（不在本任務範圍，但記下來）**：
- 在 `recordPricingInterest()` 外面包 try/catch，dbPut 失敗也 toast「已記錄」（先 UI 反饋、資料失敗事後 retry）
- 或：先 toast，再背景 dbPut

#### 文案改寫優先級

- **對齊產品路線圖**：✅ 符合 Phase 1「測收費意願」目的
- **對齊商業評估**：✅ 商評第 325 行就建議「在設定頁加『未來打算收費，您的意見』按鈕」
- **UX 影響**：低（單行文案 + 按鈕標籤）
- **實作成本**：極低（改 2 個字串）
- **建議優先級**：🔥 這週做（順手連 bug 一起修）
- **理由**：使用者原話「未來打算收費，你贊成的話點 Yes」比現在「未來打算收費，您的意見？/ 告訴我想法」更直接、更能驅動點擊。連同 try/catch 修一起做，3 行內可解
